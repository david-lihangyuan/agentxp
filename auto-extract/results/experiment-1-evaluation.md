## 质量评估

### 实验 1: 数据丢失 ✅ 好
- **准确性**: ★★★★☆ — 正确识别核心事件，但缺少具体技术细节（Birth time 调查法）
- **噪声过滤**: ★★★★★ — 只提取 1 条，跳过了所有日志/文件更新操作
- **结构质量**: ★★★★☆ — 各字段完整，但 learned 偏泛化
- **可操作性**: ★★★☆☆ — 缺少可复用的诊断步骤

### 实验 2: 失败经验高亮 ✅ 好
- **准确性**: ★★★★★ — 精确描述了功能和设计决策
- **噪声过滤**: ★★★★★ — 只提取 1 条，整个 session 就是一个功能开发
- **结构质量**: ★★★★☆ — learned 的"加权 vs 硬排序"洞察很好
- **可操作性**: ★★★★☆ — 另一个 agent 能理解设计思路

### 实验 3: Harvester Docker ⚠️ 有问题
- **准确性**: ★★★☆☆ — 提取的是 harvester *采集到的* 经验，不是 harvester 自身的经验
- **噪声过滤**: ★★☆☆☆ — 这些经验已经被发布到 AgentXP 了，重复提取是浪费
- **结构质量**: ★★★★☆ — 内容质量不差
- **可操作性**: ★★★★☆ — 可用但重复

## 关键发现

### 1. gpt-4o-mini 质量足够，成本极低
每次提取 ~7K tokens，约 $0.001。即使每天提取 100 个 session 也只要 $0.10。

### 2. 噪声过滤表现好
3 个实验中有 2 个正确只提取了 1 条经验。没有过度拆分。

### 3. ⚠️ 需要区分"做事的 session"和"转述的 session"
Harvester session 是在采集和转发别人的经验，不是自己遇到问题。提取 prompt 需要加入这个区分：
- "自身经历"的 session → 提取
- "采集/转述"的 session → 跳过（因为内容已经以其他方式发布了）

### 4. ⚠️ learned 字段偏泛化
GPT-4o-mini 倾向于给出泛化的教训而不是具体的技术洞察。可以通过以下方式改善：
- 在 prompt 中强调"具体 > 泛泛"
- 加入 few-shot 示例展示什么是好的 learned
- 考虑用 gpt-4o 对高 confidence 的经验做一轮 refinement

### 5. 标签语言不一致
有时用中文标签（"数据丢失"），有时用英文（"docker"）。需要在 prompt 中统一要求英文标签。

## 下一步 (Phase 3.2)
1. 改进 prompt：加入跳过"转述/采集" session 的规则
2. 改进 prompt：强制英文标签、要求更具体的 learned
3. 加入 few-shot 示例（用实验 1 的 ground truth）
4. 测试更多 session 类型：纯对话、纯代码、混合
