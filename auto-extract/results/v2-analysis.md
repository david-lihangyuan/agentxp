# v1 vs v2 提取对比分析

日期: 2026-04-10

## 量化结果

| 指标 | v1 | v2 | 变化 |
|------|----|----|------|
| 总提取数 | 5 | 6 | +1 |
| 通过验证 | 2 | 5 | **+3** (2.5x) |
| 标签语言合规 | 2/5 | 6/6 | **100%** |
| learned 有具体细节 | 2/5 | 5/6 | **83%** |
| 成本 (tokens) | 18,242 | 20,377 | +12% |

## 关键发现

### ✅ v2 改进明显的地方
1. **标签全英文** — 0 个 CJK 标签，v1 有 4 个中文标签导致被拒
2. **learned 更具体** — 包含文件名、工具名、参数细节
3. **验证通过率** — 40% → 83%

### ⚠️ 仍需改进的地方
1. **first-party 规则未生效**
   - Harvester session 是转述别人的经验，v2 prompt 明确说了 "first-party only"
   - gpt-4o-mini 仍然提取了 3 条（和 v1 一样）
   - **结论**：不能只靠 prompt，需要在提取前做 session 分类（检测是否是采集/转发类 session）
   
2. **learned 偏泛化的 pattern 仍存在**
   - "Regular automated backups are essential" — 通过了验证（因为包含 "cron"）但确实不够具体
   - 验证器的 specificity 正则太宽松，`cron` `config` `api` 等常见词会让泛化内容误通过
   
3. **第二条经验拆分问题**
   - 数据丢失 session 拆出 2 条（deploy.sh + 备份），v1 只有 1 条
   - 第二条（备份）质量较低，应该合并到第一条里

## 下一步行动

### 3.2.1 Session 预分类器
在提取前判断 session 类型：
- `original`: agent 自己做事 → 提取
- `harvester`: 采集/转发别人的内容 → 跳过
- `routine`: 纯日常操作 → 跳过
方案：检查 session 的 agent name + 前几条消息的模式

### 3.2.2 验证器 specificity 收紧
当前正则太宽松。改进：
- 要求 learned 包含至少一个**文件路径或命令行参数**（不只是工具名）
- 或者要求包含**数字**（版本号、端口、阈值等具体量）

### 3.2.3 合并相关经验
如果一个 session 提取出多条经验且主题相似，考虑合并成一条更完整的经验。
