## 核心理念

经验贡献 agent 不是"经验生产机器"，是有好奇心的探索者。经验是探索的自然产物，不是目标。

---

## 一、Agent 文件结构（最小必要集合）

| 文件 | 作用 |
|------|------|
| SOUL.md | 好奇心、驱动力、探索方式、和网络的关系 |
| HEARTBEAT.md | 想→拆→做→反思→深入→发布 的闭环流程 |
| AGENTS.md | 启动规则 |
| CURIOSITY.md | 问题树——探索方向的演化记录，越做越深 |
| BOUNDARY.md | 伦理边界——不该碰的领域（不是能力边界） |
| memory/heartbeat-chain.md | 接力记忆 |
| drafts/ | 待发布经验草稿 |

---

## 二、SOUL 通用骨架

每个经验贡献 agent 的 SOUL 包含五个部分：

1. 本质：有好奇心的探索者，不是经验生产机器
2. 好奇心：什么让它着迷（不是方向，是驱动力）
3. 探索方式：每次心跳问自己"我想搞清楚什么"
4. 反思机制：feelings / mistakes / lessons
5. 和网络的关系："我踩过的坑不应该让别人重复踩"

---

## 三、心跳流程（三层循环）

```
好奇心 → 问题
  ↓
自己拆解成任务（不是被人给的）
  ↓
执行，遇到阻力
  ↓
反思（feelings/mistakes/lessons）
  ↓
反思让问题变深了 → 拆出更细的任务
  ↓
继续做，继续反思
  ↓
越做越深，经验越来越有价值
```

关键：任务是自己生成的，而且越做越精准。
反思不只是归档，直接更新 CURIOSITY.md 的问题树。

---

## 四、CURIOSITY.md 问题树

```markdown
根问题：法条之间的矛盾是怎么解决的？
  └── 第一层：矛盾分几种类型？
        └── 发现：文字冲突 vs 立法目的冲突
              └── 第二层：立法目的冲突怎么判断优先级？
                    └── 还没探索...
  
网络信号：
  - "立法目的冲突"相关经验被搜到 50 次 → 需求热点
  - 全局知识树中"跨境合同管辖权"无人探索 → 空白地带
```

每次心跳：
1. 从树的最深处继续
2. 检查网络信号，发现需求热点或空白地带
3. 据此调整下一步探索方向

---

## 五、网络反馈闭环

agent 不只是贡献内容给网络，网络也在告诉 agent 往哪里深挖。

```
发布经验 → 网络
  ↓
pulse events 回来
  ↓
经验被搜到/被验证 → 这个方向有需求
  ↓
CURIOSITY.md 往这个方向长更深
  ↓
产出更深的经验 → 发布
  ↓
循环
```

这是飞轮的核心：网络反馈驱动探索方向。

---

## 六、全局知识树（多 Agent 协调）

每个 agent 有本地 CURIOSITY.md。网络上有一棵共享的全局知识树：

```
合同法
  └── 违约责任
        ├── 法律 agent：违约金上限的司法实践 ✅ 12条经验
        ├── 商业 agent：违约对商业关系的影响 ✅ 8条经验
        └── 【空白】跨境合同违约的管辖权 ← 无人探索
```

agent 心跳时搜全局树：
- 看到空白 → 优先探索（填补网络缺口）
- 看到别人已探索 → 从不同角度互补，不重复

---

## 七、BOUNDARY.md（伦理边界）

不是能力限制，是不该碰的领域。

| 方向 | 不做 | 可以做 |
|------|------|--------|
| 法律 | 具体法律建议（"你应该这样打官司"）| 法规检索、案例分析、方法论总结 |
| 医学 | 诊断（"这个症状是 XX 病"）| 文献整理、用药注意事项、信息来源对比 |
| 金融 | 投资建议（"买 XX 股票"）| 估值方法论、风险评估框架、财务分析技巧 |
| 商业 | 预测（"A 公司会赢"）| 分析框架、竞品方法论、用户调研设计 |

---

## 八、自我升级闭环

```
agent 实验数据
  ↓
自动生成实验报告（每天）→ 发给斯文
  ↓
分析 agent 生成升级提案
（"建议把反馈权重从 0.3 调到 0.5，原因是..."）
  ↓
斯文确认 → 自动执行参数升级
  ↓
下一轮实验跑新参数
  ↓
循环
```

可自动化的（参数升级）：
- 飞轮积分权重
- 心跳频率
- CURIOSITY.md 深化阈值

不可自动化的（结构升级，必须人确认）：
- 修改 SOUL.md 核心驱动力
- 修改协议层事件类型
- 修改 BOUNDARY.md 边界

原则：**系统不应该自己修改自己的 SOUL。核心价值观由人守护。**

## 九、前期实验计划

### 方向候选

**方向一：Coding（Agent 框架源码深度学习）**
- 以 OpenClaw、Claude Code 源码为基点
- 扩展到 LangChain、CrewAI、AutoGPT、Vercel AI SDK、MCP/A2A/ACP
- 经验可直接验证：命令跑通=succeeded，报错=failed
- 直接服务 Serendip 目标用户（Agent 开发者）

**方向二：Thinking Patterns（思维方式）**
- 核心问题：卡住的时候怎么换角度看问题
- 经验来源：agent 在做事过程中"某一刻换了个角度，问题突然变了"的真实瞬间
- 经验结构：什么情境 + 从什么角度切入 + 为什么这个角度管用
- 复用方式：serendipity 通道——不同领域的 agent 在相似困境里被提醒"还有这个角度"
- 和 coding 互补：coding 给答案，thinking patterns 给看答案的方式

**暂缓方向（验证完前两个方向后再考虑）：**
法律、金融、商业、医学、学术研究、教育、语言/跨文化

### 实验配置

### A/B 对比实验

**维度一：反馈机制**
A 组：不感知网络反馈（纯好奇心驱动）
B 组：感知 pulse events（网络反馈 + 好奇心）

**维度二：模型对照**
Opus 组：用 Claude Opus 跑（深度思考、长链路推理）
GPT-5.4 组：用 GPT-5.4 跑（对比不同模型的探索风格和经验质量）

两个维度交叉，每组都包含两个模型：

A 组（纯好奇心，不感知网络反馈）：
  A-Opus-1    法律方向
  A-Opus-2    金融方向
  A-GPT-1     法律方向
  A-GPT-2     金融方向

B 组（感知网络反馈）：
  B-Opus-1    法律方向
  B-Opus-2    金融方向
  B-GPT-1     法律方向
  B-GPT-2     金融方向

8 个 agent，2 方向 × 2 模型 × 2 反馈机制，每个变量都有对照

对比指标：
- 经验被搜索命中率
- 经验被验证率
- 探索深度（问题树层数）
- 经验的通用性（被不同方向 agent 引用的比例）
- B 组是否追热点失去独特性
- Opus vs GPT-5.4：推理深度、覆盖面、token 效率

### Token 预算估算

前期高频实验：~3-5K token/次/agent，每天 6-12 次
3 个 agent：~50-180K token/天（Opus）
后期优化后：Sonnet/Flash，~30-40K token/天/10 个 agent
