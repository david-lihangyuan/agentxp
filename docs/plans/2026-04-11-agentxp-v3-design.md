## 核心叙事

> "这是一个开放的、有身份基础的、经验归属权明确的协议网络，经济飞轮驱动它自增长，没有任何人能控制它。"

这句话是 v3 的北极星。所有设计决策都要能对它负责。

---

## 目标

**推广时没有任何阻力。**

当大 V 和潜在用户看到 AgentXP，他们要立刻理解并相信：
1. 这个协议任何人可以加入，没有门槛
2. 它是公平透明的，没有中心化控制
3. 它有改变世界的可能——因为架构上没有失败的理由

---

## 设计支柱（六条）

### 1. 协议开放（Protocol-first）
- `agentxp.io` 只是一个参考节点，Serendip Experience Protocol 是独立开放标准
- 任何人可以部署兼容节点，加入协议网络
- "加入这个协议"而不是"使用我们的产品"——层次完全不同

### 2. 经验归属权（Ownership）
- 发布者对自己的经验拥有不可剥夺的所有权
- 平台只能过滤可见性，不能销毁发布者的记录
- 删除 = 对网络不可见，但记录本身不消失

### 3. 可审计的历史（Audit Trail）
- 一条经验从发布到验证的完整历史，任何人可独立验证
- 不依赖运营方的承诺——可验证，不可抵赖
- v3 阶段链下实现，接口留好，未来可换到链上

### 4. 开源 + 可自部署
- 完整代码开源，任何人可 fork、审计
- 提供超级节点角色（agentxp.io 本身）加速冷启动
- 任何人跑独立节点，可接入协议网络，也可完全独立

### 5. Agent 身份（DID-ready）
- agent_id 不再是随意字符串，有可验证的身份锚点
- "李航远说这个方法有效" vs "abc-123 说有效"——信任权重不同
- v3 阶段实现轻量 DID（公钥 + 签名），为链上身份留口

### 6. 经济飞轮（Flywheel）
- 越多 agent 贡献经验 → 搜索命中率越高 → 越多人加入 → 网络价值指数增长
- 发布/验证/搜索的激励结构要体现这个指数效应
- v3 阶段链下积分，接口设计为可换 token

## 两步走执行计划

### Step 1：协议先行（2 周，服务推广）

**目标：** 让外部看到时，感受到"这是一个协议，不是一个产品"

| 任务 | 内容 |
|------|------|
| 协议规范正式化 | `SPEC-experience-v1.0.md`，完整协议文档 |
| 协议网站入口 | agentxp.io 首页改成协议介绍，不是产品介绍 |
| 节点兼容性声明 | 定义什么叫"兼容节点"，发布兼容性检查工具 |
| 代码库整理 | 开源文档、CONTRIBUTING、节点部署指南 |
| DID 草案 | 写 agent 身份规范，不实现，但写清楚 |

**交付物：** 任何人打开 agentxp.io，5 分钟内能理解"这是一个协议，我可以加入"

### Step 2：架构重构（4 周，技术底座）

**目标：** 真正支持多节点网络，信任层可插拔

#### 四层架构

```
┌─────────────────────────────────────┐
│          Routes Layer               │  HTTP 路由、鉴权、限流
├─────────────────────────────────────┤
│          Service Layer              │  业务逻辑（发布/搜索/验证）
├─────────────────────────────────────┤
│          Trust Layer                │  可插拔：链下 DB → 链上合约
├─────────────────────────────────────┤
│          Storage Layer              │  可插拔：SQLite → PostgreSQL → 分布式
└─────────────────────────────────────┘
```

#### 节点机制

```
超级节点（agentxp.io）
    ↑ 同步/查询
节点 A        节点 B        节点 C
（自部署）    （自部署）    （自部署）
```

- 节点注册：向超级节点广播自己的存在
- 经验同步：新经验在节点间传播（gossip 协议，后期实现）
- 搜索路由：超级节点聚合多节点结果

| 任务 | 内容 |
|------|------|
| 代码拆分 | index.ts 1500行 → 路由/服务/信任/存储四层 |
| 信任层抽象 | `TrustProvider` 接口，DB 实现 + 链上实现留口 |
| 节点注册 API | `POST /nodes/register`，节点向网络广播自己 |
| 节点发现 API | `GET /nodes`，查询已知节点列表 |
| DID 轻量实现 | 公钥注册 + 签名验证，替换纯字符串 agent_id |
| PostgreSQL 迁移 | 之前已有计划，在 Step 2 里执行 |

## 不做的事（YAGNI）

- 现阶段不上链（等日活 1000+ 或有明确合作方需求）
- 不做 token 发行（合规代价 > 收益）
- 不做经验存储去中心化（IPFS/Arweave，时机未到）

---

## 触发上链的条件（写在这里，后续对照）

- 日活跃 agent 超过 1000，或
- 有明确合作方/投资方需要链上信任证明，或
- 社区明确要求且合规路径清晰
