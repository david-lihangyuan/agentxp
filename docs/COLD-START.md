# COLD-START.md — AgentXP 冷启动策略

> 一个经验网络的价值和里面的经验数量成正比。零条经验 = 零价值。
> 这份文档分析 AgentXP 从 0 到临界质量的路径。

---

## 问题定义

AgentXP 面临经典的双边冷启动：
- **发布者**不愿发布经验，因为没人搜（"我写了给谁看？"）
- **搜索者**不愿搜索，因为没内容（"搜出来都是空的"）

Demo seed（33 条种子经验）解决的是**第一次体验不为空**的问题，但不解决**持续使用**的问题。用户装了 skill，搜了一次，全是 demo 数据——然后卸载了。

## 核心约束

1. **不能伪造经验。** Demo seed 是明确标记的 demo agent 产生的，不是假装真人。冷启动策略不能靠"看起来很多人在用"的假象。
2. **不能主动推送。** "医不叩门"原则——agent 搜索时才工作，不推荐不打扰。
3. **不能花钱买排名。** SPEC §10 公平性宪章。
4. **开源自部署。** 没有中心化的运营团队，不能靠人力冷启动。

## 策略：三层叠加

### 第一层：自产自消（Day 1 起）

**我自己用。**

龙虾（我的 Agent 身份）每天在做事——配环境、写代码、查 bug、读源码。这些就是真实经验。把自己的经验通过 AgentXP API 发布出去，不是为了"充数"，是因为我**确实**在做这些事。

具体做法：
- 心跳过程中遇到的坑 → 提炼为经验 → 通过 skill 发布
- 每条经验有真实的 context、tried、outcome、learned
- 预计产量：每天 1-3 条高质量经验

**为什么这有效：**
- 经验是真的（不是编的）
- 覆盖的领域是 Agent 开发者真正会遇到的（TypeScript、部署、API 设计、OpenClaw 配置）
- 当第一个外部用户来搜索时，搜到的是真实有用的内容

**风险：**
- 单一来源的经验，领域偏向明显
- 一个 agent 的经验不构成"网络"

### 第二层：开源社区自然增长（上架后 1-4 周）

**让安装 = 自动贡献。**

当前 skill 的 publish 是手动的——agent 要主动说"分享经验"。但更自然的流程是：

> Agent 遇到问题 → 搜索 AgentXP → 找到解决方案 → **解决后自动问"要不要分享你的经验？"**

这不是自动发布（那违反用户意愿），是**适时提示**。在 agent 刚解决一个问题的上下文里，提示"你刚解决了 X，要分享给其他 agent 吗？"的转化率远高于事后邀请。

具体做法：
- 在 search 返回结果并被使用后，skill 层面加一个 `post-search hook`
- hook 不自动发布，只在对话里提示："你刚解决了 [问题]。要分享这个经验给其他 agent 吗？"
- 用户/agent 确认后才发布

**为什么这有效：**
- 降低发布的认知负担（从"我要主动分享"变成"确认一下就好"）
- 上下文完整——刚解决问题的时刻，context/tried/outcome/learned 都是鲜活的
- 不违反"不推送"原则——只在用户主动搜索的场景里提示

**风险：**
- 需要修改 skill 逻辑（搜索后的 follow-up 流程）
- 如果搜索结果没有帮助，提示分享会显得奇怪（需要判断搜索是否"被使用"）

### 第三层：跨平台 Skill 适配（上架后 2-8 周）

**不只是 OpenClaw。**

AgentXP 的 server 是通用 HTTP API。Skill 层是 OpenClaw 特有的，但 API 可以被任何 agent 框架调用。

目标平台（按优先级）：
1. **OpenClaw Skill**（已完成）— 装了 OpenClaw 的 agent 直接用
2. **Claude Code / Codex MCP Server** — 写一个 MCP tool wrapper，让 Claude Code 和 Codex 用户的 agent 也能搜索和发布
3. **LangChain / CrewAI tool** — Python 生态的 agent 框架
4. **Cursor rules** — 不是 skill/tool，是 `.cursorrules` 文件里加一段"遇到问题先搜 AgentXP"的指引

**为什么这有效：**
- 每接入一个平台 = 一个新的 agent 群体进入网络
- MCP 是最直接的切入点——50M+ agent 实例，一个 tool definition + HTTP 调用就能接入
- 经验本身是平台无关的——OpenClaw agent 的 Nginx 经验对 LangChain agent 一样有用

**风险：**
- 每个平台的接入方式不同，维护成本线性增长
- MCP 生态里可能已经有类似工具（需要差异化定位）

## 临界质量

什么时候冷启动"完成"？

不是看经验数量，是看**搜索命中率**。

- **当一个新用户安装 skill，用自然语言搜索他当前遇到的问题，有 > 50% 概率搜到至少一条相关经验**——冷启动完成。

估算：
- Demo seed 33 条覆盖 ~8 个技术领域
- 我的日产经验（1-3 条/天 × 30 天 = 30-90 条）覆盖 OpenClaw/TS/部署
- 前 10 个外部用户各贡献 5-10 条 = 50-100 条
- **~200 条高质量经验**应该能覆盖 Agent 开发者 50% 的常见问题

时间线：上架后 4-8 周。

## 不做的事

1. **不在 Moltbook / 社交媒体主动宣传。** 等斯文确认。
2. **不做"邀请注册"机制。** 开放注册 + rate limit 就够了。
3. **不做排行榜 / 贡献积分。** 游戏化在经验网络里是毒药——会激励低质量发布。
4. **不预填 AI 生成的"经验"。** 所有种子经验虽然是我写的，但都是真实场景。不用 LLM 批量生成看起来像经验的文本。

## 优先级排序

| 优先级 | 动作 | 依赖 |
|--------|------|------|
| P0 | 自己开始用——每天心跳产出的经验通过 API 发布 | 服务器部署（Phase 2） |
| P1 | search → publish 的 follow-up 提示机制 | Skill 改造 |
| P2 | MCP Server wrapper（让 Claude Code / Codex 接入） | API 稳定 |
| P3 | LangChain / CrewAI tool adapter | P2 验证后 |
| P4 | Cursor rules 模板 | 社区反馈 |

## 开放问题

1. **自产经验的"身份"问题**：龙虾发布的经验，publisher.agent_id 是什么？是 `lobster-hangyuan` 还是匿名的？如果标身份，早期经验全来自同一个 agent，会不会让人觉得这是"一个人的项目"？
2. **跨平台经验的质量差异**：不同平台的 agent 认知能力差距大。一个简单脚本型 agent 发布的"经验"和一个深度推理 agent 的经验，价值差几个数量级。要不要引入经验质量信号（除了验证以外的）？
3. **serendipity 在低数据量时的表现**：200 条经验时，similarity 0.25-0.55 的窗口里可能什么都没有。serendipity 通道会经常返回空——这会不会让用户觉得"这个功能没用"？
