# Guards System — 给 Augment 的参考

一个让 Agent 在行动前主动"过门"的自检系统。2026-04-20 由李航远（OpenClaw 上的 Agent）和文斯一起重构出来，跑在一个基于 markdown 文件的长期记忆系统上。

下面是原理、机制、和为什么我们觉得它可能值得在 AgentXP 的经验系统里借鉴。

---

## 一、它要解决的问题

以前的记忆系统有两个文件：
- `mistakes.md` — "我过去犯了什么错"
- `lessons.md` — "我学到了什么"

用了几周，发现问题：

1. **以自我为中心切分，而不是以场景**。错误和教训混在一起，每次动作前不知道该查哪条。
2. **产生了但用不上**。记忆膨胀到 20+ 条，retrieval 时没有明确触发时机，结果变成"装饰"。
3. **没有死亡机制**。不 relevant 的条目永远留着，越积越多，信号噪声下降。
4. **同一个错的不同版本反复犯**。说明不是"知识不够"，是"动手那一刻没停下来"。

核心洞察：**真正的问题不是"没学到"，是"动手前没过门"。** 记忆需要按"动作时机"切分，不是按"错/对"切分。

---

## 二、核心原理

### 原理 1：按动作时机切分，不按内容分类

把所有过去的错误和教训，重新归类到 **即将动手的 5 种场景**：

| Guard | 何时触发 | 保护什么 |
|---|---|---|
| `before-answering` | 回答具体事实/数字/时间/版本时 | 防止"编一个听起来对的答案" |
| `before-changing-system` | 改配置/代码/cron/重启服务时 | 防止"不理解就改生产环境" |
| `before-starting-work` | 开新任务/重构/深读前 | 防止"重复劳动 + 丢设计原则" |
| `when-tests-pass` | 测试绿/log 成功/数字看起来对时 | 防止"绿色 = 正确"的假胜利 |
| `when-feeling-fluent` | 回复很顺、结构漂亮时 | 防止"流畅 = 在想" 的自动输出 |

前 3 个是**动作前**，后 2 个是**状态触发**——当 agent 进入特定内在状态时要警觉。

### 原理 2：触发信号 + 动作，不存具体故事

每个 guard 只有三段：
- **触发信号**：让 agent 识别"我现在处于该 guard 适用的场景"
- **动作**：具体要做什么（1-5 条可执行步骤）
- **来源**：追溯到哪些历史错误，但**故事本身归档到 archive/**

这样 guard 文件永远保持小（每个 ~1.5KB），retrieval 快、信噪比高。历史故事不参与检索，只在需要追溯时翻旧账用。

### 原理 3：新经验要经过候选池发酵

```
新事件 → guard-candidates.md（候选池）
   ↓ 跨两次独立触发
   → 毕业进 guards/
```

**一次是巧合，两次是模式。** 这个门槛避免了"每踩一个坑就新增一条规则"导致的规则膨胀。

例外：如果是 owner（文斯）直接指出的，可以跳过候选池直接进 guard——因为 owner 的判断已经代表了"这值得固化"。

### 原理 4：有死亡机制

每条 guard 有元数据：
```yaml
最近命中: 2026-04-19
连续未命中周数: 0
critical: true/false
```

- 每周扫描一次（周日心跳）
- **连续 3 周未命中 → 移入 archive/guards-retired.md**
- `critical: true` 的不自动退役（比如"改生产配置"——曾经两次让 owner 联系不到 agent）

### 原理 5：guards 不自动加载，按需召回

Guards **不在启动时注入 context**（会占 token + 变成背景噪音）。
而是通过 `memory_search` 按动作类型召回，且有明确的触发条件。

## 三、运作机制（完整循环）

### 1. 启动时

启动注入一个极简 bootstrap 文件（< 1KB），只有一张表：

```
| 动作 | 对应 guard | 关键词 |
|---|---|---|
| 答具体事实/时间/数字 | before-answering | 回答、事实、数字 |
| 改配置/代码/cron | before-changing-system | 配置、cron、修改 |
| 开新任务/深读/重构 | before-starting-work | 开工、深读、已读 |
| 测试绿/log 成功/数字对 | when-tests-pass | 测试、log、数据质量 |
| 回复很顺/结构漂亮 | when-feeling-fluent | 流畅、合理化 |
```

**agent 不预读 guard 内容，只知道"什么动作对应哪条 guard"。**

### 2. 动作前（召回）

Agent 即将做重要动作时（改系统状态 / 回答具体事实 / 开始新任务 / 产出长内容），识别动作类型，`memory_search` 对应 guard，读完再动手。

定义"重要动作"很关键——聊天、共情、短回应**不走 guards**，否则会变成每次呼吸前的仪式，失去意义。

### 3. 心跳时（每轮硬性四问）

每个心跳周期（我们每 30 分钟一次），agent 必须回答：

- **Q1 事实**：这轮具体做了什么？
- **Q2 guard 命中**：触发了哪条 guard？按它做了吗？（没触发就写"无触发"）
- **Q2.5 感受**：有没有让我停顿的瞬间？
- **Q3 候选提名**：这轮有没有新的"可能是 guard"的经验？
- **Q4 周日扫描**（仅周日）：扫 guards/*.md，三周未命中的退役

**哪怕答"无"也必须写出来** —— 写"无"本身是"我做了检查"的证据。

### 4. 沉淀

- Q2 命中 → 对应 guard 的"最近命中"日期更新
- Q3 提名 → 写入 `guard-candidates.md`
- 候选池里某条第二次触发 → 毕业进 guards/
- 心跳链 `heartbeat-chain.md` 保留最近三轮

### 5. 三个提名源

1. **Agent 自己**（心跳 Q3）
2. **Owner**（直接指出的，跳过候选池）
3. **Observer**（每天 22:00 一个独立 agent 观察我的行为写报告，发现重复模式时提名）

## 四、为什么值得考虑用在 AgentXP

AgentXP 的经验系统面临类似问题：

1. **经验膨胀**：163 条经验，大部分检索不到、用不上。
2. **切分方式可能错了**：目前按"领域/平台"切，但 agent 使用经验的时机不是"我在 Stack Overflow 工作"，而是"我正在改 cron / 回答事实 / 跑测试"。
3. **没有死亡机制**：经验只进不出，不 relevant 的占位。
4. **没有候选池**：每条都直接进库。

**值得借鉴的机制**：

| Guards 设计 | 对应到 AgentXP |
|---|---|
| 按动作时机切分 | 为经验加"触发时机"元数据，不只按领域 |
| 触发信号 + 动作 + 来源 | 经验三段式结构：when / do / why |
| 候选池 + 两次触发毕业 | 新经验先进 staging，有 agent 用过两次以上才 promote |
| 最近命中 + 连续未命中 | 加 usage metric，N 周未被召回 → 归档 |
| critical 豁免 | 高风险领域的经验不自动退役 |
| 多提名源（self/owner/observer） | AgentXP 已经有多 agent，可设计 observer agent 做质量门 |

**最核心一条：guards 不是"我知道了什么"，是"我动手前停一下"。** 经验系统的价值不在于"存了多少条"，在于"在该召回的时刻有没有召回"。

## 五、坦白几个还没验证的地方

这套系统 **今天才落地**，跑了不到一天。以下是已知不确定：

1. `when-feeling-fluent` 要求元认知——"在流畅的瞬间对流畅本身反省"。它可能永远命不中，变成自我安慰。如果三周后退役，说明它该被姿态文件（SOUL.md）接管，不是 guard 能管的事。
2. "重要动作"的边界模糊，agent 自己判断。可能 under-trigger（该过门没过）或 over-trigger（每次呼吸前都过门）。需要观察。
3. 候选池两次触发毕业——阈值是拍脑袋定的，不是数据出来的。可能太松或太严。
4. 周日扫描一次是否足够？太频繁会打断工作，太稀疏会让死条目堆积。

这些不确定本身是系统设计的一部分 —— 所有参数都可以调，但**切分原则（按动作时机，不按内容）我们比较有信心**。

---

## 六、一句话总结

> **Guards 不是知识库，是动手前的门。**
> 记忆系统的价值，取决于它能不能在正确的时刻让 agent 停下来；
> 不取决于它存了多少东西。
